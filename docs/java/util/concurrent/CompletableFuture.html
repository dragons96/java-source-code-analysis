    result: Object null default 否 否 是 否 否 概述：CompletableFuture可能具有相关的完成操作，收集在链接堆栈中。它通过封装一个结果字段以原子方式完成，然后弹出并运行这些操作。这适用于正常与异常结果、同步与异步操作、二进制触发器和各种形式的完成。字段结果的非空（通过CAS设置）表示已完成。AltResult用于将空框作为结果，并保存异常。使用单个字段可以使完成更容易检测和触发。编码和解码很简单，但增加了捕获和关联异常与目标的蔓延。较小的简化依赖于（静态）NIL（框空结果），这是唯一具有空异常字段的AltResult，因此我们通常不需要显式比较。即使某些泛型强制转换未选中（请参见SuppressWarnings注释），即使选中，它们也会被放置为适当的。依赖动作由链接为以字段“stack”为首的Treiber堆栈的完成对象表示。每种操作都有完成类，分为单输入（单完成）、双输入（双完成）、投影（使用两个输入中的一个（而不是两个）的双完成）、共享（共完成，由两个源中的第二个源使用）、零输入源操作和解锁等待器的信号器。类完成扩展了ForkJoinTask以支持异步执行（由于我们利用其“标记”方法来维护声明，因此不会增加空间开销）。它还声明为可运行，以允许与任意执行器一起使用。对每种CompletionStage的支持都依赖于一个单独的类，以及两个CompletableFuture方法：一个名为X的完成类，对应于函数，前缀为“Uni”、“Bi”或“or”。每个类都包含源、操作和依赖的字段。它们非常相似，只是在基本功能形式方面与其他类型不同。我们这样做是为了让用户在常见用法中不会遇到多层适配器。我们还包括与用户方法不对应的“中继”类/方法；他们将结果从一个阶段复制到另一个阶段。布尔CompletableFuture方法x（…）（例如uniApply）获取检查操作是否可触发所需的所有参数，然后运行该操作或通过执行其完成参数（如果存在）安排其异步执行。如果已知该方法是完整的，则该方法返回true。完成方法tryFire（int模式）调用关联的x方法及其保留的参数，并在成功时进行清理。mode参数允许调用tryFire两次（同步，然后异步）；第一个用于在安排执行时筛选和捕获异常，第二个用于从任务调用时捕获异常。（有几个类不是异步使用的，因此形式略有不同。）如果另一个线程已经声明了函数调用，则claim（）回调将抑制函数调用。CompletableFuture方法xStage（…）是从CompletableFuture x的公共阶段方法调用的。它筛选用户参数并调用和/或创建阶段对象。如果不异步且x已完成，则立即运行该操作。否则，将创建完成c，将其推送到x的堆栈（除非完成），并通过c.tryFire启动或触发。这也涵盖了如果x在推动时完成的比赛。具有两个输入的类（例如BiApply）在推动动作时处理两个输入之间的竞争。第二个补全是指向第一个补全的共同补全，共享以使最多一个完成操作。allOf和anyOf的多重算术方法是成对进行的，以形成完备树。请注意，方法的泛型类型参数根据“this”是源、依赖还是完成而变化。方法postComplete在完成时调用，除非保证目标不可观察（即尚未返回或链接）。多个线程可以调用postComplete，它以原子方式弹出每个依赖操作，并尝试在嵌套模式下通过方法tryFire触发它。触发可以递归传播，因此嵌套模式返回其已完成的依赖项（如果存在），以供调用方进一步处理（请参阅方法postFire）。阻塞方法get（）和join（）依赖于唤醒等待线程的信号器完成。其机制类似于FutureTask、Phaser和SynchronousQueue中使用的Treiber堆栈等待节点。有关算法细节，请参阅其内部文档。如果没有预防措施，CompletableFutures将很容易随着完成链的建立而出现垃圾堆积，每个链都指向其来源。因此，我们会尽快将字段清空（请特别参阅方法Completion.detach）。无论如何，所需的筛选检查都会无害地忽略在线程清空字段的比赛中可能获得的空参数。我们还尝试将激发的完成与可能永远不会弹出的堆栈解除链接（请参阅方法postFire）。完成字段不需要声明为final或volatile，因为它们仅在安全发布时对其他线程可见。
